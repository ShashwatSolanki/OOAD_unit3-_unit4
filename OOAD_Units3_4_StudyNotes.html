1. GRASP Principles

GRASP = General Responsibility Assignment Software Patterns. Published by Craig Larman. 9 fundamental principles for assigning responsibilities to classes/objects in OO design. A responsibility is a contract/obligation a class must fulfil.

Types of Responsibility

Consequences

  • Doing: Creating objects, doing calculations, initiating actions in other objects, coordinating activities
  • Knowing: Knowing private encapsulated data, related objects, things it can derive/calculate

1. Creator

Problem: Who creates a new instance of class X?
Solution: Assign creation to the class that contains, aggregates, records, closely uses, or has initialising data for X.

Container creates Contained. Sale creates SalesLineItems. Board knows Squares, so Board creates Square in Monopoly.
Creator â€” "Container creates Contained". Factory pattern is used for complex creation situations.

2. Information Expert

Problem: What is the basic principle to assign responsibilities?
Solution: Assign responsibility to the class that has the information needed to fulfil it â€” objects do things related to the information they have.

ClassResponsibility (POS Example)
SaleKnows sale total
SalesLineItemKnows line-item subtotal
ProductDescriptionKnows product price
Information Expert â†’ assign to class that has the data. Board is information expert for Square positions in Monopoly.

3. Low Coupling

Coupling = degree of dependency between classes.
Problem: How to support low dependency, low change impact, high reuse?
Solution: Assign responsibilities so that coupling remains low.

  • Tight Coupling: Class creates instances of another, directly accesses its methods/fields â†’ hard to change, hard to reuse
  • Loose Coupling: Classes interact through interfaces/abstractions â†’ changes in one class don't cascade
Low Coupling: use interfaces + Dependency Injection. Subclassing = high coupling. TypeX has attribute of TypeY = coupled.

4. High Cohesion

Cohesion = measure of how strongly related and focused the responsibilities of a class are.
Problem: How to keep complexity manageable?
Solution: Assign responsibilities so cohesion remains high.

Low cohesion problems: Hard to understand, hard to reuse, hard to maintain, brittle.

High cohesion = class does ONE focused thing. Low cohesion = does many unrelated things.

5. Controller

Problem: Who should be responsible for handling a system event (input from UI layer)?
Solution: Assign responsibility to either:

  • FaÃ§ade Controller: Class representing the overall system/device/subsystem
  • Use Case Controller: Class representing a specific use case scenario (SaleHandler)

Controller coordinates activity but delegates work â€” it doesn't do the work itself.

Bloated Controller (Bad): Single class receiving all system events, doing too much work, storing too much data.

Controller: first object that receives message from UI layer. Two types: FaÃ§ade Controller and Use Case Controller.

6. Polymorphism

Problem: How to handle alternatives based on type?
Solution: Assign responsibility using polymorphic operations to types where behaviour varies.

7. Indirection

Problem: How to avoid direct coupling between two components while keeping reuse ability?
Solution: Assign responsibility to an intermediate class that mediates between them.

Related patterns: Adapter, Bridge, Mediator. Example: Salary class as intermediary between caller and Employee.

Indirection = intermediate class for mediation. Avoids direct coupling. Related: Adapter, Bridge, Mediator.

8. Pure Fabrication

Problem: What class should have responsibility when assigning only to domain objects would violate High Cohesion or Low Coupling?
Solution: Assign responsibilities to an artificial convenience class that does not represent a domain concept.

Examples: utility classes, service classes, helper classes, DAO (Data Access Object).

Monopoly: a Cup class holds dice and manages rolling â€” not a real domain entity but solves the design problem cleanly. Can be reused in any dice-based game.
Pure Fabrication = artificial class, not from domain model. Created when domain objects would cause poor cohesion/coupling.

9. Protected Variations (PV)

Problem: How to design so variation/instability in elements doesn't cause undesirable impact on others?
Solution: Identify points of predicted variation â†’ create a stable interface around them.

  • Variation point: Branching point in existing system or requirements
  • Evolution point: Possible future branching point not yet in requirements

Similar to Open-Closed Principle. Example: Account interface with different account types implementing it â€” BankSystem doesn't change when a new account type is added.

2. SOLID Principles

SOLID = 5 design principles by Robert C. Martin (Uncle Bob). Acronym coined by Michael Feathers. Goal: flexible, stable software with low coupling + high cohesion.

Problems SOLID solves (Bad Design symptoms)

SymptomDescription
RigidityEvery change causes cascade of changes in dependent modules
FragilitySoftware breaks in unrelated places on every change
ImmobilityUseful modules can't be reused â€” too many dependencies
ViscosityIt's easier to do the wrong thing (hacks) than the right design

3. SRP â€” Single Responsibility Principle

"A class should have one, and only one, reason to change." â€” Robert C. Martin

  • Each class does only one thing
  • Can be applied to classes, components, microservices
  • Ensures low coupling, fewer test cases, easier maintenance

Violation Example

// BAD: Book has two responsibilities â€” data + inventory search
class Book {
    String title, author;
    String getTitle() { return title; }
    void setTitle(String t) { title = t; }
    void searchBook() { ... } // â† WRONG: inventory logic in Book!
}

Fixed (SRP applied)

// GOOD: Book = data only. InventoryView = search only.
class Book {
    String title, author;
    String getTitle() { return title; }
    void setTitle(String t) { title = t; }
}

class InventoryView {
    Book book;
    InventoryView(Book b) { this.book = b; }
    void searchBook() { ... }
}
SRP: one class = one reason to change. Advantage: fewer test cases, lower coupling, better organisation.

4. OCP â€” Open/Closed Principle

"Open for extension, Closed for modification." â€” Bertrand Meyer

Add new features by extending code (via inheritance/interfaces), never by modifying existing working code.

Violation â†’ Fix

// VIOLATION: Adding new discount type requires modifying DiscountManager
class DiscountManager {
    void processCookbookDiscount(CookbookDiscount d) { ... }
    void processBiographyDiscount(BiographyDiscount d) { ... } // â† must modify!
}

// GOOD: Use an interface â€” DiscountManager never changes for new types
interface BookDiscount { String getBookDiscount(); }

class CookbookDiscount implements BookDiscount {
    @Override
    public String getBookDiscount() { return "30% between Dec 1-24"; }
}

class DiscountManager {
    void processBookDiscount(BookDiscount d) { ... } // â† never needs to change!
}

If OCP is violated: entire functionality must be retested, SRP also breaks, maintenance overhead increases.

OCP: extend via inheritance/interfaces, don't modify. Violating OCP also breaks SRP.

5. LSP â€” Liskov Substitution Principle

"Derived types must be completely substitutable for their base types." â€” Barbara Liskov (1987)

A child class should never change characteristics of its parent. Subclasses should do at least as much as the parent, never less.

Applies to is-a (inheritance) hierarchies. Avoids overuse/misuse of inheritance.

Violation

// AudiobookDelivery CAN'T deliver to physical locations
// but it MUST override getDeliveryLocations() â€” VIOLATION!
class AudiobookDelivery extends BookDelivery {
    @Override
    void getDeliveryLocations() { /* can't be implemented */ } // â† LSP violation!
}

Fix â€” Extra abstraction layer

class BookDelivery  { String title; int userID; }

class OfflineDelivery extends BookDelivery  { void getDeliveryLocations() { ... } }
class OnlineDelivery  extends BookDelivery  { void getSoftwareOptions()   { ... } }

class HardcoverDelivery extends OfflineDelivery { @Override void getDeliveryLocations() { ... } }
class AudiobookDelivery extends OnlineDelivery  { @Override void getSoftwareOptions()   { ... } }
// âœ“ Now AudiobookDelivery never has to deal with physical location methods!
LSP: is-a relationship. Substitute subclass for superclass â†’ behaviour unchanged. Child must NOT do LESS than parent.

6. ISP â€” Interface Segregation Principle

"Clients should not be forced to implement interfaces they do not use."

Break fat (bloated) interfaces into smaller, client-specific interfaces. Many small specific interfaces > one general-purpose interface.

Xerox case: one large Job class with all responsibilities â†’ segregated into multiple task-specific interfaces.

Violation â†’ Fix

// FAT interface â€” HardcoverUI must implement listenSample() which it doesn't need!
interface BookAction { void seeReviews(); void searchSecondhand(); void listenSample(); }

// GOOD: Split into specific interfaces
interface BookAction      { void seeReviews(); }
interface HardcoverAction extends BookAction { void searchSecondhand(); }
interface AudioAction     extends BookAction { void listenSample(); }

class HardcoverUI implements HardcoverAction { ... } // only what it needs
class AudiobookUI implements AudioAction     { ... } // only what it needs
ISP: no forced implementation of unused methods. Fat interface â†’ split into role interfaces. Many small > one large.

7. DIP â€” Dependency Inversion Principle

"Program to the interface, not the implementation."

  • High-level modules should NOT depend on low-level modules â€” both should depend on abstractions
  • Abstractions should NOT depend on details â€” details should depend on abstractions
  • Combination of OCP + LSP (Robert C. Martin)

Violation â†’ Fix

// BAD: Shelf directly depends on Book (concrete class)
class Shelf { Book book; void addBook(Book b) { ... } }  // â† can't add DVD!

// GOOD: Introduce Product abstraction
interface Product  { void seeReviews(); void getSample(); }
class Book implements Product { ... }
class DVD  implements Product { ... }

class Shelf {
    Product p;
    void addProduct(Product product) { ... } // â† works for Book, DVD, Magazine...
}

// DIP in action: OrderService depends on Database interface, not MySQL
interface Database { void saveOrder(Order o); }
class OrderService {
    private Database db; // injected â€” could be MySQL, MongoDB, etc.
    OrderService(Database db) { this.db = db; }
    void placeOrder(Order o) { db.saveOrder(o); }
}
DIP: both high-level and low-level depend on abstractions. Goal: avoid tightly coupled code. Related: OCP + LSP.

9. Singleton Pattern (Creational)

Intent: Ensure a class has only one instance, and provide a global access point to it.

Motivation: Some services must behave consistently from one shared point. If multiple logger/configuration objects exist, system state becomes inconsistent.

Applicability: Use when exactly one object should control a shared service such as configuration, cache manager, logger, or thread pool manager.

Real analogy: A country has only one official government.

Use cases: Logging, caching, thread pool, database connection, configuration manager.

Solution Steps

  1. Make constructor private (prevent external instantiation)
  2. Add a private static field to store the single instance
  3. Create a public static getInstance() method that returns the cached instance

Method 1: Lazy Instantiation (NOT thread-safe)

class Singleton {
    private static Singleton obj;
    private Singleton() {}  // private constructor
    public static Singleton getInstance() {
        if (obj == null) obj = new Singleton();  // lazy: created only when needed
        return obj;
    }
}

Method 2: Synchronized (thread-safe, slower)

public static synchronized Singleton getInstance() {
    if (obj == null) obj = new Singleton();
    return obj;
}  // â† overhead: synchronizes every call

Method 3: Eager Instantiation (thread-safe)

private static Singleton obj = new Singleton();  // created when class loads

Method 4: Double-Checked Locking (Best âœ…)

private static volatile Singleton obj = null;  // volatile ensures visibility
public static Singleton getInstance() {
    if (obj == null) {
        synchronized (Singleton.class) {  // lock only if null
            if (obj == null) obj = new Singleton();
        }
    }
    return obj;
}

Consequences

  • Controlled single access point to a shared service
  • Useful when one common object must be reused everywhere
  • Can make testing and dependency management harder if overused
4-Mark Exam Practice: A system creates multiple logger objects and separate logs appear for the same application. Identify the pattern and write a correct Java implementation.
Singleton: private constructor + static getInstance(). Use volatile for double-checked locking. Violates SRP (solves 2 problems).

10. Factory Method Pattern (Creational)

Intent: Define an interface for creating an object, but let subclasses decide which class to instantiate. Also known as: Virtual Constructor.

Client accesses objects through a common interface â€” creation details are hidden.

Motivation: Direct creation using concrete classes couples the client tightly to product classes. Any new type forces client changes.

Applicability: Use when a class should defer creation to subclasses, or when exact product types are decided by configuration or context.

Participants

RoleDescription
ProductInterface for objects the factory creates
ConcreteProductImplements Product interface
CreatorDeclares factoryMethod() returning a Product
ConcreteCreatorOverrides factoryMethod() to return a ConcreteProduct

Easy Java Example

interface Notification { void send(); }

class EmailNotification implements Notification {
    public void send() { System.out.println("Email sent"); }
}

abstract class NotificationFactory {
    abstract Notification createNotification();
}

Consequences

  • Removes direct dependency on concrete products
  • Supports extension through new creators and products
  • Introduces extra classes compared to simple object creation
4-Mark Exam Practice: A payment client directly creates concrete payment classes. Rewrite the design using Factory Method and identify the participants.
Factory = Virtual Constructor. Subclasses decide class to instantiate. Client uses common interface. Hides creation logic.

11. Builder Pattern (Creational)

Intent: Separate the construction of a complex object from its representation so the same construction process can create different representations.

Helps avoid "telescoping constructors" (constructor with 10 optional parameters).

Motivation: A complex object with many optional parts makes constructors long, unreadable, and error-prone.

Applicability: Use when object creation must happen step by step and different final products can be built using similar construction steps.

Participants

RoleDescription (Burger example)
BuilderAbstract interface for creating parts (BurgerBuilder)
ConcreteBuilderImplements building steps (CheeseBurgerBuilder)
DirectorControls the build process (Chef)
ProductThe complex object under construction (Burger)

Easy Java Example

class Burger { String bun, patty, topping; }

interface BurgerBuilder {
    void addBun();
    void addPatty();
    void addTopping();
    Burger getBurger();
}

Consequences

  • Keeps construction logic separate from the product object
  • Improves readability for objects with many optional fields
  • May be unnecessary for very small/simple objects
4-Mark Exam Practice: Explain why Builder is better than a constructor with many parameters for creating a configurable laptop or burger object.
Builder: Director controls build, ConcreteBuilder implements steps, Product = result. Avoids telescoping constructors.

12. Prototype Pattern (Creational)

Intent: Create new objects by copying existing objects (cloning), without making code dependent on their classes.

Used when object creation is expensive/complex, or when classes are specified at runtime.

In Java: uses clone() method + Cloneable interface.

Motivation: If creating and initializing an object is costly, cloning a ready object is often faster and simpler.

Applicability: Use when objects differ only slightly from existing ones or when prototypes must be registered and copied dynamically.

Key Concepts

  • Prototype Registry: Centralized store of pre-defined prototype objects for retrieval and cloning
  • Client asks prototype to clone itself â€” doesn't know the specific class
  • Deep copy required when object has references to other objects (use clone() or deserialization)

Benefits

  • Adding/removing products at runtime
  • Specifying new objects by varying values or structure
  • Reduced subclassing
  • Configuring application with classes dynamically

Easy Java Example

class IDCard implements Cloneable {
    String name;
    public IDCard clone() throws CloneNotSupportedException {
        return (IDCard) super.clone();
    }
}

Consequences

  • Reduces repeated expensive initialization
  • Supports runtime configuration through prototype objects
  • Deep copy handling can be tricky when nested references exist
4-Mark Exam Practice: A certificate generator creates many similar objects with only student details changing. Explain why Prototype is suitable and distinguish shallow copy from deep copy.
Prototype: copy via clone(). Uses Cloneable interface. Prototype Registry = cache of prototype objects. Reduces subclassing.

13. Adapter Pattern (Structural)

Intent: Convert the interface of a class into another interface that clients expect. Allows classes with incompatible interfaces to work together.

Also known as: Wrapper.

Motivation: Reusing an existing class is difficult when its interface does not match what the client expects.

Applicability: Use when you want to reuse a legacy or third-party class without modifying its source code.

Two Types

TypeMechanismKey Difference
Class AdapterMultiple inheritance (extends Adaptee)Works only for specific Adaptee class (not subclasses). Can override Adaptee behaviour.
Object AdapterComposition (holds Adaptee reference)Works for Adaptee AND its subclasses. More flexible. More delegation code needed.

Easy Java Example

interface Printer { void print(); }

class OldPrinter {
    void printOldFormat() { System.out.println("Old printer output"); }
}

class PrinterAdapter implements Printer {
    private OldPrinter oldPrinter = new OldPrinter();
    public void print() { oldPrinter.printOldFormat(); }
}

Consequences

  • Improves reuse of incompatible or legacy classes
  • Protects client from adaptee interface mismatch
  • Adds one wrapper layer and delegation code
4-Mark Exam Practice: A client expects a Printer interface but only an old class with printOldFormat() exists. Draw the adapter structure and write the code.
Adapter = Wrapper. 4 participants: Target, Adapter, Adaptee, Client. Object adapter uses composition; Class adapter uses inheritance.

14. Facade Pattern (Structural)

Intent: Provide a unified interface to a set of interfaces in a subsystem. Defines a higher-level interface that makes the subsystem easier to use.

Think: fopen() â€” one call that hides complex file system operations.

Motivation: A subsystem may expose too many classes and steps. Clients become hard to write and tightly coupled to internal details.

Applicability: Use when you want one simple entry point to a complex subsystem such as booking, startup, reporting, or home theatre control.

Consequences

  • Shields clients from subsystem components â€” fewer objects to deal with
  • Promotes weak coupling between subsystem and clients
  • Can eliminate complex/circular dependencies
  • Example: Home theatre â€” one watchMovie() method controls amplifier, projector, screen, popcorn maker

Easy Java Example

class HomeTheatreFacade {
    void watchMovie() {
        System.out.println("Projector ON");
        System.out.println("Sound system ON");
        System.out.println("Movie started");
    }
}
4-Mark Exam Practice: A movie system needs separate calls to projector, speakers, and lights. Explain how Facade simplifies client code and sketch the Java class.
Facade = simplified interface to complex subsystem. Promotes weak coupling. Real example: fopen(), booking package system.

15. Proxy Pattern (Structural)

Intent: Provide a surrogate or placeholder for another object to control access to it. Client doesn't know about the proxy â€” it appears as the real object.

Motivation: Access to the real object may need lazy loading, protection, caching, or remote communication.

Applicability: Use when access must be controlled without changing the interface seen by the client.

3 Main Types of Proxy

TypePurposeExample
Remote ProxyLocal representative for object in different address spaceRemote method invocation
Virtual ProxyCreates expensive objects on demand (lazy init)Image not loaded until displayed
Protection ProxyControls access rights to objectCorporate proxy server, access control
Cache ProxyCaches results to improve performanceCaching YouTube video on first load

Participants

  • Subject: Interface exposing functionality to clients
  • Real Subject: Actual implementation hidden behind proxy
  • Proxy: Hides real object, clients talk to this
Real-world example: Corporate network proxy checks requests, allows/blocks websites before passing to server.

Easy Java Example

interface Image { void display(); }

class ProxyImage implements Image {
    public void display() { System.out.println("Load image only when needed"); }
}

Consequences

  • Adds control, validation, caching, or lazy loading transparently
  • Keeps client code unchanged because proxy and real subject share interface
  • Introduces an additional forwarding layer
4-Mark Exam Practice: An expensive image should load only when the user opens it. Identify the proxy type and write the class outline.
Proxy = surrogate/placeholder. Virtual=lazy init, Remote=different address space, Protection=access control, Cache=memoization.

16. Flyweight Pattern (Structural)

Intent: Use sharing to support large numbers of fine-grained objects efficiently. Share common state between many objects to save RAM.

Motivation: Systems with thousands of similar objects waste memory if each object stores the same internal data repeatedly.

Applicability: Use when many objects share intrinsic state and differ mainly in external context.

Key Concept: Intrinsic vs Extrinsic State

StateDescriptionExample (Document Editor)
IntrinsicIndependent of context, shareable, stored IN flyweightCharacter code (letter 'A')
ExtrinsicDependent on context, NOT shareable, passed in by clientPosition, font, color (passed per use)

Participants

  • Flyweight: Interface to receive/act on extrinsic state
  • ConcreteFlyweight: Implements Flyweight, stores intrinsic state (sharable)
  • UnsharedConcreteFlyweight: Not all flyweights need sharing (e.g., Row, Column)
  • FlyweightFactory: Creates and manages shared flyweight objects â€” returns existing if available
  • Client: Computes/stores extrinsic state, gets flyweights from factory
Java String Constant Pool is a classic Flyweight. All "hello" literals point to the same object. Browsers: image loaded once, reused from cache.

Easy Java Example

class Letter {
    private char symbol;   // intrinsic state
    Letter(char s) { symbol = s; }
    void display(int row, int col) { System.out.println(symbol + " at " + row + "," + col); }
}

Consequences

  • Saves memory by sharing intrinsic state
  • Useful for editors, strings, caches, and repeated visual objects
  • Separating intrinsic and extrinsic state increases design complexity
4-Mark Exam Practice: A document editor stores millions of repeated character objects. Explain how Flyweight reduces memory and distinguish intrinsic from extrinsic state.
Flyweight: share intrinsic state. Intrinsic = shareable/context-independent. Extrinsic = not shared/context-dependent. FlyweightFactory manages pool.

17. Bridge Pattern (Structural)

Intent: Decouple an abstraction from its implementation so that the two can vary independently.

Motivation: If every abstraction is subclassed for every implementation, the number of classes grows too fast. Example: BasicRemoteSonyTV, BasicRemoteSamsungTV, AdvancedRemoteSonyTV, AdvancedRemoteSamsungTV.

Applicability: Use when you have two independent dimensions of variation, such as remote type and TV brand, message type and sending platform, or shape and drawing API.

Structure

ParticipantRole
AbstractionHigh-level view used by client (Remote)
RefinedAbstractionExtended abstraction (AdvancedRemote)
ImplementorInterface for implementation side (Device)
ConcreteImplementorActual implementations (TV, Radio)

Mini UML (Class Diagram)

Interaction Idea

Client calls remote.togglePower(). Remote delegates the work to its device. The abstraction and implementation can both be extended independently.

Easy Java Example

interface Device {
    void on();
    void off();
}

class TV implements Device {
    public void on()  { System.out.println("TV ON"); }
    public void off() { System.out.println("TV OFF"); }
}

class Remote {
    protected Device device;
    Remote(Device d) { device = d; }
    void togglePower(boolean on) { if (on) device.on(); else device.off(); }
}

Consequences

  • Avoids class explosion caused by combining multiple dimensions through inheritance
  • Improves extensibility on both sides
  • Adds one more indirection layer, so design becomes slightly more abstract
Bridge uses composition, not inheritance, to separate abstraction from implementation. Remote-Device is the classic memory trick.

18. Composite Pattern (Structural)

Intent: Compose objects into tree structures to represent part-whole hierarchies so clients can treat individual objects and groups uniformly.

Motivation: A client should not need separate code for one file and a folder of files. The same operation like showDetails() should work on both.

Applicability: File systems, organisation charts, menu trees, GUI component trees.

Structure

ParticipantRole
ComponentCommon interface for leaf and composite
LeafPrimitive object with no children
CompositeStores children and forwards operations to them
ClientTreats all objects through Component interface

Mini UML (Class Diagram)

Easy Java Example

interface FileSystemItem { void show(); }

class FileItem implements FileSystemItem {
    private String name;
    FileItem(String n) { name = n; }
    public void show() { System.out.println(name); }
}

class Folder implements FileSystemItem {
    private List<FileSystemItem> children = new ArrayList<>();
    public void add(FileSystemItem item) { children.add(item); }
    public void show() {
        System.out.println("Folder");
        for (FileSystemItem item : children) item.show();
    }
}

Consequences

  • Simplifies client code because leaf and composite are treated uniformly
  • Makes recursive structures easy to model
  • Can make it harder to restrict which objects can contain children
Composite is best remembered as tree structure plus uniform treatment of single object and group.

19. Decorator Pattern (Structural)

Intent: Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending behaviour.

Motivation: If you create subclasses for every feature combination, the class count becomes huge. Example: PlainCoffee, MilkCoffee, SugarCoffee, MilkSugarCoffee.

Applicability: Use when you want to add features at runtime without changing the original class.

Structure

ParticipantRole
ComponentCommon interface (Coffee)
ConcreteComponentBase object (SimpleCoffee)
DecoratorWraps a Component reference
ConcreteDecoratorAdds new behaviour/state (MilkDecorator)

Mini UML (Class Diagram)

Easy Java Example

interface Coffee { String description(); }

class SimpleCoffee implements Coffee {
    public String description() { return "Plain coffee"; }
}

class MilkDecorator implements Coffee {
    private Coffee coffee;
    MilkDecorator(Coffee c) { coffee = c; }
    public String description() { return coffee.description() + " + milk"; }
}

Consequences

  • More flexible than static inheritance
  • Lets us stack responsibilities in many combinations
  • Introduces many small wrapper objects
Decorator and Adapter both wrap objects, but Decorator adds behaviour while Adapter changes interface.

20. Chain of Responsibility Pattern (Behavioural)

Intent: Avoid coupling sender of a request to its receiver by giving multiple objects a chance to handle the request. Chain receiving objects and pass request along until one handles it.

Motivation: The sender should not need to know exactly which object will process the request.

Applicability: Use when multiple handlers are possible and the request should travel until one accepts it.

Participants

  • Handler: Interface â€” receives request, dispatches to chain, knows first handler only
  • ConcreteHandler: Handles request if responsible, else forwards to successor
  • Client: Initiates request to first handler in chain

Transaction Approval (Code)

abstract class TransactionProcessor {
    private TransactionProcessor successor;
    abstract protected Double getApprovalLimit();
    public void approve(Transaction t) {
        if (t.getAmount() <= getApprovalLimit())
            System.out.println("Approved by " + getDesignation());
        else if (successor != null)
            successor.approve(t);   // pass to next in chain
    }
}

// Manager (limit 1L), VP (limit 10L), CEO (limit 25L)
// Chain: manager â†’ vicePresident â†’ CEO
manager.setSuccessor(vicePresident);
vicePresident.setSuccessor(ceo);

Consequences

  • Reduces coupling between request sender and receiver
  • Lets us add or reorder handlers easily
  • A request may remain unhandled if no object accepts it
4-Mark Exam Practice: A leave request may be approved by team lead, manager, or director depending on duration. Identify the pattern and write the handler chain.
Chain of Responsibility: avoids coupling sender-receiver. Request may fall off end if unhandled. Used in Java exception handling.

21. Command Pattern (Behavioural)

Intent: Encapsulate a request as an object, allowing parameterisation of clients, queuing requests, and support for undo/redo.

Motivation: Requests often need to be stored, queued, logged, or undone instead of executed immediately.

Applicability: Use when menu items, buttons, jobs, or remote actions should be represented as separate objects.

Participants

RoleDescriptionStock Example
CommandInterface with execute()Order (interface)
ConcreteCommandBinds Receiver + action, implements execute()BuyStock, SellStock
ClientCreates ConcreteCommand, sets ReceiverCommandPatternDemo
InvokerAsks Command to carry out request (stores commands)Broker
ReceiverKnows how to perform the operationStock

Full Stock Trading Example

interface Order { void execute(); }

class BuyStock implements Order {
    private Stock stock;
    BuyStock(Stock s) { stock = s; }
    public void execute() { stock.buy(); }
}

class Broker {  // Invoker
    private List<Order> orderList = new ArrayList<>();
    public void takeOrder(Order o) { orderList.add(o); }
    public void placeOrders() {
        for(Order o : orderList) o.execute();
        orderList.clear();
    }
}

Consequences

  • Separates invoker from receiver
  • Makes queuing, logging, and undo operations easier
  • Introduces extra command classes
4-Mark Exam Practice: A remote control should support TV on/off commands and later undo. Explain why Command fits and identify client, invoker, receiver, and command.
Command: Invoker asks command to execute. Receiver does actual work. ConcreteCommand binds Receiver+action. Supports undo via rollback.

22. Interpreter Pattern (Behavioural)

Intent: Define a grammar for a language and provide an interpreter to deal with that grammar.

Use when you need to interpret sentences in a domain-specific language (DSL).

Motivation: Some problems are naturally expressed as small languages or rule sets that must be parsed and evaluated.

Applicability: Use for simple grammars such as arithmetic expressions, filtering rules, or mini command languages.

Participants

  • AbstractExpression: Declares interpret() operation
  • TerminalExpression: Implements interpret() for terminal symbols (leaf nodes)
  • NonterminalExpression: One per grammar rule, recursive interpret()
  • Context: Contains global information for interpreter
  • Client: Builds abstract syntax tree (AST), invokes interpret()

Example: Math Expression

// 2 + 3 modelled as:
new AddExpression(new ConstantExpression(2), new ConstantExpression(3))

// (2 + 6) + (1 + 9):
new AddExpression(
    new AddExpression(new ConstantExpression(2), new ConstantExpression(6)),
    new AddExpression(new ConstantExpression(1), new ConstantExpression(9))
)

Applications: Regular expressions, SQL query parsers, mathematical expression evaluators, programming language interpreters.

Consequences

  • Makes grammar structure explicit in code
  • Easy to extend for small languages by adding new expression classes
  • Can become class-heavy and inefficient for large grammars
4-Mark Exam Practice: Explain how the expression (2 + 6) + (1 + 9) is represented using Interpreter pattern objects.
Interpreter: grammar + AST. Context = global info. TerminalExpression = leaf. Client builds tree + calls interpret(). Used for regex, SQL, DSLs.

23. Iterator Pattern (Behavioural)

Intent: Provide a way to access elements of an aggregate object sequentially without exposing its underlying implementation.

Provides a uniform way to traverse any collection (List, Set, Map) regardless of its internal structure.

Motivation: The client should traverse collections without knowing whether the data is stored in an array, list, tree, or custom structure.

Applicability: Use when traversal logic should be separate from collection implementation.

Participants

RoleDescription
IteratorInterface: hasNext(), next(), remove()
ConcreteIteratorImplements Iterator, tracks current position
AggregateInterface for creating Iterator object (createIterator())
ConcreteAggregateImplements Aggregate, returns proper ConcreteIterator

Advantages

  • Single Responsibility Principle â€” traversal logic extracted to separate class
  • Open/Closed Principle â€” add new collections/iterators without changing client code
  • Can iterate same collection in parallel (each iterator has its own state)
  • Supports lazy evaluation (retrieve elements only when needed)

Easy Java Example

List<String> names = new ArrayList<>();
names.add("A");
names.add("B");

Iterator<String> it = names.iterator();
while(it.hasNext()) System.out.println(it.next());
4-Mark Exam Practice: Why is Iterator preferred over exposing the internal array of a collection? Explain with Java collection traversal.
Iterator: sequential access without exposing implementation. ConcreteIterator tracks position. Java: Iterator interface, for-each loops.

24. Observer Pattern (Behavioural)

Intent: Define a one-to-many dependency so that when one object changes state, all its dependents are notified automatically.

Motivation: Hard-coding updates from one object to many other objects creates tight coupling. A subject should not know concrete observer classes.

Applicability: Event systems, stock tickers, GUI listeners, YouTube subscriptions, weather updates.

Structure

ParticipantRole
SubjectStores observers, attaches/detaches, sends notifications
ObserverDefines update() method
ConcreteSubjectMaintains state of interest
ConcreteObserverReceives updated information

Mini UML + Interaction

Easy Java Example

interface Subscriber { void update(String video); }

class Channel {
    private List<Subscriber> subs = new ArrayList<>();
    void subscribe(Subscriber s) { subs.add(s); }
    void upload(String video) {
        for (Subscriber s : subs) s.update(video);
    }
}

Consequences

  • Loose coupling between subject and observers
  • Easy to add new observers without changing subject logic
  • Notification order and excessive updates can become tricky
Observer = publish-subscribe style notification. One subject, many observers.

25. Strategy Pattern (Behavioural)

Intent: Define a family of algorithms, encapsulate each one, and make them interchangeable.

Motivation: Long if-else or switch blocks for multiple algorithms violate OCP. Example: different payment modes.

Applicability: Sorting choices, discount policies, route selection, payment methods.

Structure

ParticipantRole
StrategyCommon algorithm interface
ConcreteStrategySpecific implementation
ContextUses a Strategy object

Mini UML

Easy Java Example

interface PaymentStrategy { void pay(int amount); }

class UpiPayment implements PaymentStrategy {
    public void pay(int amount) { System.out.println("Paid by UPI: " + amount); }
}

class Checkout {
    private PaymentStrategy strategy;
    void setStrategy(PaymentStrategy s) { strategy = s; }
    void makePayment(int amount) { strategy.pay(amount); }
}

Consequences

  • Removes conditional logic from context
  • Supports runtime switching of algorithms
  • Client must understand available strategies
Strategy changes the algorithm. State changes the behaviour based on internal state. Both use composition, but intent is different.

26. State Pattern (Behavioural)

Intent: Allow an object to alter its behaviour when its internal state changes. The object appears to change its class.

Motivation: Huge conditionals based on current state are hard to maintain. Example: vending machine behaviour changes in idle, has-money, and sold state.

Applicability: Workflows, game characters, vending machines, network connection lifecycle.

Structure

ParticipantRole
ContextMaintains current state object
StateCommon behaviour interface
ConcreteStateImplements behaviour for one state

Mini State Transition

Idle --insertCoin--> HasMoney --pressButton--> Dispensing --done--> Idle

Easy Java Example

interface State { void handle(); }

class IdleState implements State {
    public void handle() { System.out.println("Insert coin"); }
}

class VendingMachine {
    private State state = new IdleState();
    void setState(State s) { state = s; }
    void request() { state.handle(); }
}

Consequences

  • Localises state-specific behaviour into separate classes
  • Removes large state conditionals
  • Can increase number of classes
State is ideal when the same request behaves differently depending on internal state.

27. Mediator Pattern (Behavioural)

Intent: Define an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by keeping objects from referring to each other explicitly.

Motivation: If every object talks directly to every other object, dependencies become a web. A mediator centralises communication.

Applicability: Chat rooms, air traffic control, GUI dialog coordination.

Structure

ParticipantRole
MediatorDeclares communication operations
ConcreteMediatorCoordinates colleagues
ColleagueCommunicates through mediator

Mini UML

Easy Java Example

class ChatRoom {
    static void showMessage(String user, String msg) {
        System.out.println(user + ": " + msg);
    }
}

class User {
    private String name;
    User(String n) { name = n; }
    void send(String msg) { ChatRoom.showMessage(name, msg); }
}

Consequences

  • Reduces many-to-many coupling between colleagues
  • Centralises control logic
  • Mediator itself can become too complex if overloaded
Mediator centralises communication. Observer distributes notification. Do not mix them up.

28. Memento Pattern (Behavioural)

Intent: Capture and externalise an object's internal state so that the object can be restored later without violating encapsulation.

Motivation: Undo/restore is difficult if outside classes directly access private fields.

Applicability: Text editor undo, game save points, transaction rollback snapshots.

Structure

ParticipantRole
OriginatorCreates memento and restores from it
MementoStores internal state snapshot
CaretakerKeeps memento safe, but does not inspect it

Mini UML + Flow

Editor -> createMemento() -> SaveState
History stores SaveState
Editor <- restore(SaveState) <- History

Easy Java Example

class SaveState {
    private String text;
    SaveState(String t) { text = t; }
    String getText() { return text; }
}

class Editor {
    private String text;
    SaveState save() { return new SaveState(text); }
    void restore(SaveState s) { text = s.getText(); }
}

Consequences

  • Supports undo/rollback cleanly
  • Preserves encapsulation better than exposing internals
  • Many snapshots can consume memory
Originator creates the memento, Caretaker stores it, Memento holds snapshot state.

29. Visitor Pattern (Behavioural)

Intent: Represent an operation to be performed on elements of an object structure. Visitor lets you define a new operation without changing the element classes.

Motivation: If many unrelated operations are added to stable element classes, those classes become bloated.

Applicability: Reporting, tax calculation, compiler AST processing, validation across many element types.

Structure

ParticipantRole
VisitorDeclares visit methods for each element type
ConcreteVisitorImplements one operation
ElementDeclares accept(visitor)
ConcreteElementCalls visitor.visit(this)

Mini UML

Easy Java Example

interface Item { int accept(TaxVisitor v); }

class Book implements Item {
    public int accept(TaxVisitor v) { return v.visit(this); }
}

class TaxVisitor {
    int visit(Book b) { return 5; }
    int visit(Fruit f) { return 2; }
}

Consequences

  • Easy to add new operations on existing elements
  • Groups related operations into one visitor
  • Adding a new element type requires changing visitor interface and all visitors
Visitor is good when element classes are stable but operations keep increasing.

30. Template Method Pattern (Behavioural)

Intent: Define the skeleton of an algorithm in an operation, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps without changing the algorithm structure.

Motivation: Many processes follow the same broad sequence but differ in some steps. Example: tea and coffee preparation.

Applicability: Cooking recipes, report generation, data import pipelines, framework lifecycle hooks.

Structure

ParticipantRole
AbstractClassDefines template method and common steps
ConcreteClassOverrides variable steps

Mini UML + Sequence

Beverage.prepare()
  1. boilWater()
  2. brew()           <- subclass specific
  3. pourInCup()
  4. addCondiments()  <- subclass specific

Easy Java Example

abstract class Beverage {
    final void prepare() {
        boilWater();
        brew();
        pourInCup();
        addCondiments();
    }
    abstract void brew();
    abstract void addCondiments();
}

Consequences

  • Promotes code reuse for common steps
  • Controls algorithm order from base class
  • Uses inheritance, so behaviour variation is fixed through subclassing
Template Method fixes the algorithm skeleton in parent class. Strategy swaps the whole algorithm through composition.

31. Anti-Patterns

What are Anti-Patterns?

A solution to a problem that generates more problems than it addresses. Anti-patterns describe "negative solutions" with a path for correction (refactoring).

ConceptDefinition
PatternGood ideas â€” repeatable solution to a common problem
Anti-PatternBad ideas â€” recognised poor solution with a refactored alternative
RefactoringBetter ideas â€” the corrective solution

3 Viewpoints of Anti-Patterns

  • Development Anti-Patterns: Programmer-level issues when solving programming problems
  • Architectural Anti-Patterns: System/enterprise-level structure problems (most serious)
  • Management Anti-Patterns: Software organisation and people problems

Root Causes (7 Deadly Sins)

SinEffect
HasteTight deadlines â†’ neglect important activities
ApathyNot caring about solving known problems
Narrow-MindednessRefusal to learn proven solutions
SlothAdapts simplest "solution" regardless of quality
AvariceGreed â†’ over-complex, hard-to-maintain systems
IgnoranceLack of motivation to understand things
PrideRefusing to reuse existing solutions (NIH syndrome)

Anti-Pattern 1: Analysis Paralysis (Management)

Symptom: Project continuously restarted; analysis never ends; cost exceeds expectation without predictable endpoint.

Cause: Waterfall assumption â€” insisting all analysis must complete before design starts. Goals not well defined.

Solution: Use incremental development. Internal increments (infrastructure) and external increments (user-visible features) built iteratively.

Anti-Pattern 2: Vendor Lock-In (Architecture)

Symptom: Software completely dependent on one vendor. Every product upgrade drives maintenance. Delays in promised features stall development.

Cause: Product selected on marketing alone, not technical inspection. No isolation layer between app and vendor product.

Solution: Isolation Layer â€” abstract layer separating application from vendor technology. App talks to abstraction, not vendor API directly.

Anti-Pattern 3: The Blob (Development)

Symptom: One class with 60+ attributes and operations, doing everything. Other classes only hold data. Violates SRP.

Cause: Lack of OO architecture, no enforcement of design principles.

Tiny Code Smell Example

// BAD: one blob class doing everything
class Library {
    void addBook() { }
    void searchBook() { }
    void issueBook() { }
    void payFine() { }
    void registerUser() { }
}

Refactoring steps:

  1. Identify related attributes/operations â†’ group by responsibility
  2. Find "natural homes" â†’ migrate to appropriate classes
  3. Remove far-coupled/redundant associations
  4. Migrate to derived classes where appropriate
  5. Replace transient associations with type specifiers
Blob = one class does everything (60+ methods). Causes: no OO architecture. Fix: distribute responsibilities. Analysis Paralysis = too much analysis, no action.
